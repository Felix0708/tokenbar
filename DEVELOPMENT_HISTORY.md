# TokenBar 개발·변경 기록

이 문서는 TokenBar가 어떤 문제에서 출발했고, 무엇을 어떻게 고쳤으며, 그 과정에서 아이디어가 어떻게 확장됐는지를 기록한다.

새 기능을 추가할 때는 다음 흐름으로 남긴다.

> 문제/관찰 → 원인 → 수정 내용 → 수정 이유 → 사용자에게 보이는 변화 → 다음 아이디어

## 프로젝트의 출발점

TokenBar의 출발점은 여러 AI 코딩 도구를 번갈아 사용하면서 각각의 사용량과 한도를 따로 확인해야 하는 불편함이었다.

처음 목표는 단순했다.

- 메뉴바에서 Claude와 Codex 사용량을 한눈에 보기
- 긴 로그를 직접 열지 않고 오늘 사용량과 누적 사용량 확인
- 가능하면 5시간/주간 한도도 함께 표시
- 알림 센터 위젯에서도 빠르게 확인

이후 실제 사용 과정에서 “숫자를 보여주는 것”만으로는 부족하다는 문제가 드러났다. 숫자의 출처, 조회 실패 여부, 어떤 모델을 썼는지, 현재 결제 플랜이 무엇인지까지 함께 보여줘야 사용자가 믿고 판단할 수 있었다.

## 1. Claude/Codex 메뉴바 앱의 기본 구조

### 관찰

Claude와 Codex를 사용할 때마다 웹 화면이나 CLI를 따로 열어야 했다.

### 수정

- macOS 메뉴바 앱 추가
- 알림 센터 위젯 추가
- 오늘 토큰, 누적 토큰, 예상 비용 표시
- 5시간/주간 한도 진행 바 추가
- 앱과 위젯이 공유하는 `snapshot.json` 저장소 구성

### 왜 수정했는가

사용량 확인 자체가 또 하나의 작업이 되면 앱을 계속 사용하지 않게 된다. 메뉴바와 위젯에 최소 정보만 상시 노출해 확인 비용을 줄이는 것이 핵심이었다.

## 2. Gemini 추가: 단일 공급자 앱에서 멀티 공급자 앱으로 확장

### 관찰

Claude와 Codex뿐 아니라 Gemini CLI도 함께 사용하게 되면서 서비스별 사용량을 한 곳에서 비교하고 싶어졌다.

### 수정

- Gemini 로그 파서 추가
- Gemini 모델별 토큰/비용 집계 추가
- Gemini 일일 요청 수와 수동 일일 한도 기반 추정 추가
- 표시할 공급자를 Claude/Codex/Gemini별로 켜고 끌 수 있도록 추가

### 왜 수정했는가

공급자마다 한도 API와 로그 형식이 다르기 때문에, 공통 화면은 유지하되 파싱 로직은 공급자별로 분리하는 구조가 필요했다.

### 확장된 아이디어

“토큰 카운터”에서 “여러 AI 서비스의 사용량 대시보드”로 범위가 넓어졌다. 이후 공급자가 추가되어도 각 공급자의 로컬 기록을 읽어 `ProviderUsage`로 변환하면 화면을 재사용할 수 있는 방향이 되었다.

## 3. 모델 목록을 하드코딩하지 않고 로그에서 자동 수집

### 관찰

새 모델이 나올 때마다 모델 이름을 코드에 직접 추가하면 누락되기 쉽고, 사용하지 않은 모델까지 화면에 나타났다. 반대로 모델을 펼쳤는데 일부 모델만 보이는 UI 문제도 있었다.

### 수정

- Claude/Codex/Gemini 모두 실제 로그에 등장한 모델명을 동적으로 수집
- 토큰 사용량이 있는 모델만 표시
- 모델 목록을 접었다 펼치는 UI로 변경
- 모델 수를 제목에 표시
- 모델이 많을 때 전체 목록을 스크롤할 수 있도록 변경
- 목록 높이가 한 줄로 고정되던 문제를 모델 수에 따라 조정

### 왜 수정했는가

모델 목록은 공급자 API의 “지원 모델 목록”이 아니라 사용자의 실제 사용 이력이어야 했다. 그래야 새 모델이 추가돼도 코드 수정 없이 반영되고, 사용하지 않은 모델 때문에 화면이 복잡해지지 않는다.

### 현재 동작

- 실제로 사용한 새 모델은 로그가 생긴 뒤 자동 표시
- 과거 로그가 남아 있으면 공급자가 더 이상 제공하지 않는 모델도 과거 사용 모델로 표시
- 해당 로그를 삭제하면 다음 갱신에서 목록에서 제거
- 앱 재시작 없이도 1분 주기 갱신으로 반영

## 4. “조회 불가”를 0으로 표시하지 않도록 수정

### 관찰

한도 API나 로그에 정보가 없는데도 화면에 `0%` 또는 `0토큰`처럼 보이면 실제 사용량이 0인지, 조회에 실패한 것인지 구분할 수 없었다.

### 수정

- 조회 실패 시 `–`와 설명 문구를 표시
- Claude OAuth API의 429/401/403 및 네트워크 오류 처리
- 마지막 성공값을 별도 캐시하고, 오래된 값이면 조회 시각 표시
- Gemini 로그에 토큰 필드가 없을 때 임의의 0으로 계산하지 않음
- Codex 한도는 최신 API 값이라고 과장하지 않고 마지막 세션 기록 기준으로 표시

### 왜 수정했는가

사용량 대시보드에서 가장 위험한 값은 “틀렸지만 정상처럼 보이는 값”이다. 알 수 없는 상태를 알 수 없다고 표시하는 것이 숫자를 하나 더 보여주는 것보다 중요했다.

## 5. Claude 인증 흐름을 앱 안에서 이어지게 확장

### 관찰

Claude가 로그인되지 않았을 때 사용자는 터미널을 직접 열고 `claude auth login`을 입력해야 했다. 또 키체인 접근 여부에 따라 CLI와 앱의 상태가 달라질 수 있었다.

### 수정

- `claude auth status --json` 상태 확인
- 로그인되지 않은 경우 안내 문구 표시
- 메뉴의 로그인 버튼에서 Terminal을 열고 `claude auth login` 실행
- AppleScript가 차단된 경우 명령어를 클립보드에 복사하고 Terminal 열기
- Claude 키체인과 credentials 파일을 순서대로 확인

### 왜 수정했는가

로그인 문제는 사용량 파싱 문제가 아니라 인증 상태 문제인 경우가 많다. 사용자가 원인을 찾아 헤매지 않도록 조회 실패 화면에서 바로 인증 흐름으로 이어지게 했다.

## 6. 결제 플랜 표시로 아이디어 확장

### 관찰

사용자는 토큰 비용뿐 아니라 “현재 내가 어떤 유료 플랜을 사용 중인지”도 알고 싶어 했다. 비용 추정치만 보면 구독 사용자에게 실제 청구액처럼 오해될 수 있었다.

### 수정

- Claude: `claude auth status --json`의 `subscriptionType`을 읽어 `Pro` 등 표시
- Codex: 세션의 `rate_limits.plan_type`을 읽어 `Plus` 등 표시
- 서비스 이름 옆에 플랜 배지 표시
- 스냅샷에 플랜 정보와 감지 여부 저장

### 왜 수정했는가

API 정가로 환산한 비용과 실제 구독료는 다르다. 플랜 정보를 함께 보여주면 비용이 “실제 결제액”이 아니라 “API 가격 기준 참고치”라는 맥락을 이해하기 쉬워진다.

## 7. Gemini 플랜 자동 감지의 한계와 안전한 fallback

### 관찰

Gemini CLI의 무료 계정, AI Pro, AI Ultra는 일일 요청 한도가 다르다. 하지만 현재 로컬 `google_accounts.json`에는 계정 이메일만 있고 구독 등급은 없다.

### 수정

- 구조화된 quota/plan 정보가 발견될 때만 Gemini 플랜을 자동 확정
- 자동 확정된 경우 수동 선택 대신 하나의 플랜과 일일 한도만 표시
- 정보가 없을 때는 잘못된 플랜을 추정하지 않고 수동 한도 선택을 유지
- README에 Gemini의 판별 한계를 명시

### 왜 수정했는가

이메일 주소나 `oauth-personal`이라는 인증 방식만으로 무료/Pro/Ultra를 확정하면 오판할 수 있다. 공식 CLI의 quota 정보가 없는 상태에서는 “모름”을 유지하는 것이 안전하다.

### 다음 확장 후보

- Gemini `/stats model`의 quota 결과를 안전하게 가져오는 공식 경로가 생기면 자동 감지 연결
- quota 응답의 감지 시각과 출처를 함께 표시
- 플랜이 변경되었을 때 이전 값 대신 최신 값만 표시하되, 변경 시각은 기록

## 현재 설계 원칙

1. 실제 로컬 기록을 우선한다.
2. 새 모델은 하드코딩하지 않는다.
3. 조회 실패를 0으로 위장하지 않는다.
4. 서비스별 차이는 파서에 격리하고 화면은 공통 모델을 사용한다.
5. 자동 감지가 불확실하면 수동 fallback을 제공한다.
6. 예상 비용은 실제 청구액이 아니라 참고용 추정치라고 명시한다.

## 앞으로 기록할 변경 형식

```text
## YYYY-MM-DD — 변경 제목

### 문제/관찰

### 원인

### 수정 내용

### 수정 이유

### 사용자에게 보이는 변화

```

이 문서는 기능 목록이 아니라 의사결정 기록이다. 나중에 같은 문제가 다시 생겼을 때 “왜 이렇게 구현했는가”를 다시 추측하지 않도록 유지한다.
